# FriendsAI Agent 市场调查、产品设计与实现计划（2026-02-10）

## 1. 文档目的
本文件用于沉淀当前阶段的市场分析结果，并给出可直接落地的 Agent 设计与实现计划。  
定位是“先把方向和执行路径定清楚”，后续再进入代码实现。

## 2. 结论先行（Executive Summary）
- 核心目标用户优先级：`创业销售者`（创始人、早期团队负责人、BD/销售负责人）
- 产品北极星指标：`行动转化`（Insight -> Action -> Follow-up 的闭环完成率）
- 商业化路径：`免费增值`（基础提醒免费，高价值自动化与团队协作付费）
- 渠道优先级：`飞书/微信侧`（减少切换成本，嵌入已有工作流）
- 自动化策略：`半自动执行`（系统给建议 + 用户确认关键动作）

一句话定位：  
**FriendsAI 不是“联系人数据库”，而是“把弱关系转化为有效行动”的关系经营操作系统。**

## 3. 市场调查摘要

### 3.1 用户真实任务（Job-to-be-done）
对目标用户来说，真正要完成的不是“整理通讯录”，而是：
- 在关键窗口期想起正确的人
- 用合适的理由发起低负担联系
- 持续推进关系而非一次性寒暄
- 在机会出现时能快速动员“信任网络”

### 3.2 人性与行为层面的关键洞察
1. 关系维护的最大障碍不是“不重要”，而是“行动门槛高”
- 用户知道应该联系人，但不知道如何开口、害怕打扰、怕显得功利。

2. 弱关系价值被系统性低估
- 真正带来信息与机会流动的，常是“非日常强关系”，但这类关系最容易被遗忘。

3. 时间窗口决定转化率
- 岗位变化、项目上线、行业事件、节日节点等时机，会显著提高互动成功率。

4. 关系维护需要“轻量、连续、可解释”
- 高质量行动通常是短消息、低打扰、带具体价值点，而不是长篇近况汇报。

### 3.3 竞品对比后的机会点
参考方向包含 Monica、Dex、Clay、Affinity 等关系管理产品。整体观察：
- 多数产品强在“信息聚合和记录”，弱在“可执行行动引导”
- 多数产品是 CRM 思维（管理对象），不是关系经营思维（经营信任）
- 在中国语境下，飞书/微信原生工作流整合仍有明显空白

FriendsAI 的机会：  
**从“记录关系”升级为“触发行动 + 降低心理成本 + 形成复利关系资产”。**

## 4. 隐秘但高价值的关系信号（建议重点挖掘）

### 4.1 关系债务（Relationship Debt）
定义：长期未回应、长期未触达、承诺未跟进形成的隐形负债。  
价值：比“多久没联系”更能预测关系风险。  
产品化：输出“修复优先级列表 + 推荐修复话术”。

### 4.2 互惠不平衡（Reciprocity Imbalance）
定义：用户索取信息/资源次数显著高于提供价值次数。  
价值：这是关系降温的重要先行指标。  
产品化：每次建议动作都优先给“可先提供的价值点”。

### 4.3 身份迁移窗口（Identity Transition Window）
定义：联系人岗位变化、创业、获奖、产品发布等身份变化的短周期窗口。  
价值：此时“重新连接”阻力最低，回复率更高。  
产品化：把事件驱动提醒优先级提到高于固定周期提醒。

### 4.4 关系集群桥接（Bridge Position）
定义：用户连接了两个本来不连通的圈层（例如产业方与技术社区）。  
价值：桥接节点更容易产生高价值机会。  
产品化：Agent 不只提醒“联系谁”，还提示“可以连接哪两个人”。

## 5. 产品设计：Agent 能力架构

## 5.1 目标能力分层
1. `Signal Layer`（信号层）
- 汇总联系人、会话、事件、身份变化、互动频率、承诺项。

2. `Insight Layer`（洞察层）
- 判断关系温度、机会概率、关系风险、行动优先级。

3. `Action Layer`（动作层）
- 生成可执行建议：联系对象、时机、理由、消息草稿、后续步骤。

4. `Learning Layer`（学习层）
- 追踪建议是否被采纳、是否回复、是否形成下一步，持续优化推荐。

## 5.2 Agent 角色建议（V1 -> V2）
V1（对齐当前能力并快速可用）：
- `Relationship Insight Agent`：输出关系健康与优先级（强化现有 contact_insight）
- `Action Suggestion Agent`：输出下一步动作与话术（强化现有 network_action）
- `Archive Brief Agent`：每次互动沉淀结构化记忆（沿用 archive_brief）

V2（差异化增强）：
- `Timing Agent`：捕捉身份迁移与事件窗口
- `Bridge Agent`：发现可桥接的人脉连接机会
- `Trust Maintenance Agent`：识别关系债务与互惠不平衡并给修复动作

## 5.3 关键交互原则
- 默认“建议优先”，关键外发动作必须确认（半自动）
- 每条建议必须包含：`为何是这个人`、`为何是现在`、`建议怎么说`
- 控制认知负担：每天不超过 3 条高价值建议，避免提醒疲劳

## 6. 与当前代码基线的对齐（落地前提）
当前后端已有多 capability 框架与工具确认机制雏形，适合作为 V1 基座。  
已识别的落地风险点：
- 工具确认执行链路在部分实现中仍偏 mock，需要补全真实执行与回执
- 飞书 OAuth 的 exchange/refresh 仍需完成，影响飞书侧闭环动作
- 现有能力偏“摘要和建议”，需补“结果回写与学习”数据闭环

## 7. 实现计划（不改代码版）

### 7.1 里程碑与产出

#### M0：定义与埋点基线（1 周）
目标：统一概念和指标，避免后续反复返工。  
产出：
- 关系健康、行动转化、关系债务等统一定义文档
- 事件与结果埋点规范（建议展示、采纳、发送、回复、推进）
- V1 评估看板草案

#### M1：V1 可执行闭环（2-3 周）
目标：做到“每天给出少量可执行建议，并能追踪结果”。  
产出：
- `contact_insight` 升级为可排序优先级输出（含原因）
- `network_action` 增加“时机+话术+后续动作”结构化输出
- 用户确认后动作记录与执行状态回写
- 每周关系经营简报（行动完成率、回复率、推进率）

#### M2：事件驱动与时机增强（2 周）
目标：从周期提醒升级为“窗口期触发”。  
产出：
- 事件优先级模型（岗位变化/发布/节日/历史承诺）
- 高频低价值提醒降噪策略
- “为什么是现在”解释模板

#### M3：差异化能力（2-4 周）
目标：形成难复制优势。  
产出：
- 关系债务修复策略
- 互惠平衡提示
- 桥接机会发现（介绍型 action）

### 7.2 接口与数据层计划（设计稿）

#### 能力接口（建议）
- `contact_insight`: 新增输出字段  
  - `priority_score`
  - `reason_tags[]`
  - `relationship_risk_level`
- `network_action`: 新增输出字段  
  - `timing_reason`
  - `value_first_suggestion`
  - `followup_plan`

#### 行为事件（建议新增）
- `agent_suggestion_shown`
- `agent_suggestion_accepted`
- `agent_message_sent`
- `agent_message_replied`
- `agent_followup_completed`

#### 核心数据实体（建议新增或扩展）
- `relationship_health_snapshot`
- `relationship_debt_item`
- `action_outcome_log`

## 8. 验收标准（Definition of Done）

### 8.1 产品层
- 用户每周至少收到 5 条高质量建议，其中采纳率 >= 30%
- 建议采纳后的消息回复率相对基线有显著提升
- 用户能清楚理解每条建议背后的原因（可解释性）

### 8.2 技术层
- 建议生成、确认、执行、结果回写全链路可追踪
- 飞书侧关键动作可稳定执行并记录状态
- 回归测试覆盖关键路径（建议生成、确认、失败处理）

### 8.3 业务层
- 免费用户可体验完整“建议->确认->记录”闭环
- 付费能力聚焦在“自动化深度、建议质量、团队协作”

## 9. 风险与应对
- 风险：提醒过多导致疲劳  
  - 应对：严格限量 + 价值排序 + 一键跳过低优先级
- 风险：建议泛化、像“鸡汤”  
  - 应对：每条建议绑定可验证的触发信号与行动模板
- 风险：自动化过强引发信任问题  
  - 应对：坚持半自动，关键外发动作必须确认
- 风险：渠道集成进度拖慢整体价值  
  - 应对：先做可用的建议闭环，再逐步增强自动执行

## 10. 默认策略与假设
- 默认 ICP：创业销售者
- 默认产品策略：先做“行动转化”再扩“联系人规模”
- 默认自动化边界：建议自动，外发需确认
- 默认商业化：免费增值，付费卖“更高成功率和更低操作成本”
- 默认渠道优先级：飞书/微信 > 独立后台页面
